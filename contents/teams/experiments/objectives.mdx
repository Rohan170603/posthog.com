#### Create well-configured experiments without involving the user <TeamMember name="Marcel Poelker" photo />
Ultimately we want to embed experiments in the self-driving loop, which means triggering experimentation ideas and experiment creation. Triggering ideas can have various shapes (out of thin air like scanning the codebase or by having a concrete hook like PR creation). Either way, for best user experience, the creation process should "get it right" without asking the user. This goal focuses on the 2nd step, the "getting it right" creation of experiments.

#### Skills <TeamMember name="Ruby Childs" photo />
Skills are our onboarding surface for the redefined, less-technical ICP — the MCP-facing layer that gets an experiment right the first time (~30% of experiment users already interact via MCP).

- Onboarding: help users get set up and run experiments correctly, aimed at the less-technical ICP and self-driving onboarding.
- Diagnosing: a skill/scout that inspects setup and flags errors.
    - Over MCP (config): flag and variants are sane, metric events are known, and custom exposure events are valid.
    - In a coding agent / PostHog Code (source): the flag is evaluated where the variant is shown, and the boolean-flag → experiment migration is done right.
- Educational ("Learn Experiments"): teach the concepts a non-statistical user needs to trust a result such as exposure, MDE, significance, sequential testing and common interpretation traps.
- Summarize & interpret results: a plain-language, one-page verdict on a finished experiment.
